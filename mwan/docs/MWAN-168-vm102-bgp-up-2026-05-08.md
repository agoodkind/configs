# MWAN-168 VM 102 BGP wiring (2026-05-08)

## Summary

VM 102 (opnsense-test2 on suburban) was at the
`prod-shaped-25-7-baseline-v3-2026-05-08` snapshot with a single trunk NIC on
`vmbrtrunk` and the prod-imported `config-testbed.xml` declaring WAN as
`vtnet1` with `10.250.250.2/29` plus `3d06:bad:b01:201::2/64`. There was no
`vtnet1` device because no second NIC was wired, so BGP neighbors `10.250.250.3`
and `10.250.250.4` (peers on VM 950 test-mwan) sat in `Active` and DNS via
Unbound returned `SERVFAIL`.

This slice added the second NIC `net1 virtio,bridge=vmbr2`, captured the
provider-allocated MAC, forced `configctl interface reconfigure wan` so OPNsense
bound the WAN config to the new device, and verified that BGP IPv4 came up and
installed a default route. The Tofu HCL was updated to reflect the new NIC so
`tofu plan` does not surface drift on the next refresh.

## What changed on the VM

- `qm set 102 --net1 virtio,bridge=vmbr2` added net1 with MAC `BC:24:11:D6:38:49`.
- After hot-plug, `vtnet1` came up with `IFDISABLED` and no addresses; OPNsense
  did not auto-bind the WAN config on a hot-add. `configctl interface reconfigure wan`
  via the gRPC `Exec` op fixed it: `vtnet1` then carried `10.250.250.2/29` and
  `3d06:bad:b01:201::2/64` and the WAN description.

## What worked

- `vtnet1` reachability to suburban vmbr2 stub `10.250.250.5`: 0% loss.
- `vtnet1` reachability to VM 950 transit `10.250.250.3`: 0% loss.
- BGP IPv4 to both `10.250.250.3` (test-mwan) and `10.250.250.4`
  (mwan-failover-test) came up `Established`, both peers reporting
  `GoBGP/4.4.0`. RIB has 1 entry. Neighbors are described in the FRR config as
  remote AS `4200000001`.
- `vtysh -c "show ip route"` shows `B>* 0.0.0.0/0 [200/0] via 10.250.250.3,
  vtnet1` so the kernel FIB has the BGP-learned default.
- `netstat -rn -f inet` on the OPNsense kernel side shows
  `default 10.250.250.3 UG1 vtnet1`. IPv4 default is installed.
- Egress to public IPv4 works: `ping 8.8.8.8` succeeds and `drill @8.8.8.8 pkg.opnsense.org`
  returns `89.149.222.99`.

## What did not work and why

- BGP IPv6 stayed in `Active` for both peers. The cause is a prefix overlap in
  the imported testbed config: `vlan0100` (PRIVILEGED LAN) is configured with
  `3d06:bad:b01:201::1/64` and `vtnet1` (WAN) is configured with
  `3d06:bad:b01:201::2/64`. FreeBSD installed the connected `201::/64` route
  on `vlan0100` (link#6) instead of `vtnet1` (link#10), so packets to
  `3d06:bad:b01:201::3` and `3d06:bad:b01:201::4` egress on vlan0100 (the
  trunk VLAN parent vtnet0) and never hit VM 950. NDP entries for `201::3`
  and `201::4` are `(incomplete) vlan0100`.
- Unbound recursion from `127.0.0.1` returned `SERVFAIL` for `pkg.opnsense.org`.
  Direct queries to `8.8.8.8` succeed, so this is a recursion/forwarder
  configuration issue and likely interacts with the missing IPv6 default. Did
  not investigate further in this slice.
- `pkg update` failed because Unbound did not resolve `pkg.opnsense.org`.

## Cause: imported config substitution

The generated testbed config places the same `/64` on both the WAN and the
LAN. Reading `mwan/testbed/opnsense/generated/config-testbed.xml` from a
sibling worktree (`vm102-config-import`) shows:

```
<if>vtnet1</if>
<descr>WAN</descr>
<ipaddrv6>3d06:bad:b01:201::2</ipaddrv6>
<subnetv6>64</subnetv6>
<if>vlan0100</if>
<descr>PRIVILEGED</descr>
<ipaddrv6>3d06:bad:b01:201::1</ipaddrv6>
<subnetv6>64</subnetv6>
```

In production the LAN does not collide with the test-only transit prefix.
The substitution layer needs to rewrite the WAN v6 to a testbed-only prefix
(or rewrite the LAN side) so the two interfaces do not share a `/64`. This
is a testbed-config-generator issue, not a VM 102 issue. Recording it here
so MWAN-168 owns the followup.

## Tofu HCL sync

`opentofu/vms.tf` `proxmox_virtual_environment_vm.opnsense_test2` now declares:

- `net0` on `vmbrtrunk` with the live MAC `BC:24:11:7D:6D:87`, matching what
  the provider already minted on prior import.
- `net1` on `vmbr2` with the MAC `BC:24:11:D6:38:49` from this slice's
  `qm set 102 --net1`.

`tofu plan -target=proxmox_virtual_environment_vm.opnsense_test2` reports the
network_device blocks under "6 unchanged blocks", so the HCL matches the live
NIC layout. The remaining diff is expected:

- `started = true -> false` because `started = false` is intentional in HCL
  (the operator boots VM 102 by hand or via Ansible).
- `kvm_arguments` drift because Ansible owns `args` per the MWAN-154 note.

No `tofu apply` was run.
